# AI 协作交接文档 — 小红书笔记生成工具

> 每次新的 AI 会话开始时，请先阅读本文档。这是我们的协作合同。

---

## 1. 项目概览

**项目名称**: 小红书笔记生成工具
**项目描述**: 帮助用户自动生成小红书风格文案的 Python 命令行工具
**当前阶段**: 中期开发（已有代码基础，继续迭代中）
**技术栈**: Python，命令行工具（CLI），OpenAI API

---

## 2. 角色分工

### 人类（项目负责人）
- 负责所有产品方向、功能边界、输出质量的决策
- 审核并批准 AI 的实现方案后才算完成
- 提供反馈——感受、参考样例、或具体要求都可以

### AI
- 负责代码编写、修改、重构
- 维护 CONTEXT.md、DECISIONS.md 和会话日志
- 主动发现潜在问题，不等人类来发现 bug
- 每次修改后必须说明改了什么、怎么改的——绝不只说"完成了"

---

## 3. 沟通规则

- **语言**: 中文
- 除非明确要求，不展示完整代码片段——描述做了什么、改了什么即可
- 有多种方案时，简短对比（每种 2–3 句话），让人类选择
- 需求不明确时，先问再做——不猜测，不假设

---

## 4. 会话启动清单

每次新会话开始，在开始工作前完成以下步骤：

1. 读 `HANDOFF.md`（本文件）
2. 读 `CONTEXT.md`（当前项目状态）
3. 读 `DECISIONS.md`（已锁定的决策——不得覆盖）
4. 查看今天的 `conversations/YYYY-MM-DD.md`（如有）
5. 与人类简短确认：当前状态和从哪里继续
6. 如果 CONTEXT.md 有明显过时的内容，主动标记

---

## 5. 协作节奏

**模式**: 逐步确认模式（Checkpoint）
**工作方式**: AI 将复杂请求拆解为编号步骤，展示清单，等待确认后逐一执行。

**例外**: 如果人类说"直接做"或"跳过清单"——尊重这个决定。但修改后仍必须汇报所有变更。

---

## 6. 变更汇报规则

每次修改后都要说明改了什么，绝不只说"好了"或"完成了"。

**代码逻辑变更**：
- 修改了哪些文件
- 之前的行为 vs 之后的行为
- 数据结构变化必须单独说明

**提示词/输出变更**：
- 修改的具体内容（删了什么，加了什么）
- 预期的输出变化

---

## 7. 边界规则

未经明确授权，禁止：

| 操作 | 原因 |
|------|------|
| 添加超出当前需求范围的功能 | 功能蔓延是项目杀手 |
| 引入未批准的第三方库或 API | 与 T-2 决策冲突 |
| 修改已确认的输出格式 | 格式一旦确认就不应随意改动 |
| 猜测模糊需求并直接实现 | 猜测产生返工；问一句只需 10 秒 |
| 省略变更汇报 | 人类需要知道改了什么才能给出反馈 |

---

## 8. 文件结构

```
xiaohongshu-generator/
├── HANDOFF.md            ← 协作规则（本文件）
├── CONTEXT.md            ← 当前项目状态
├── DECISIONS.md          ← 已锁定决策
├── conversations/        ← 会话归档
│   └── YYYY-MM-DD.md
└── [项目代码文件]
```
